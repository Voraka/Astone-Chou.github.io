TCP 协议分析--滑动窗口

前言

　　TCP 协议作为一个可靠的面向流的传输协议, 其流量控制由滑动窗口机制来保证. 它允许发送方在停止并等待确认前可以连续发送多个分组. 由于发送方不必每发一个分组就停下来等待确认，因此它可以加速数据的传输。

滑动窗口

　　TCP 协议在进行通信时, 接收方在返回的时候带上 ack 以及接收窗口的大小, 结合 ack 和 seq, 能够将数据分成几类:

  • 小于 ack 的数据–发送并已被确认
  • 大于 ack 小于 seq–已发送但未确认
  • 大于 seq 小于 ack + win–已在发送缓存就绪并可以发送
  • 大于 ack + win–已在发送缓存就绪但不可以发送

TCP 协议分析--Nagle

前言

　　TCP 协议中，无论发送多小数据，都要需要协议头。同时，接收方发送 ACK 的时候也需要协议头。这样， 如果网络中存在大量的小数据包就会造成网络利用率很低。为了尽可能的利用网络带宽，TCP 希望尽可能的发送足够大的数据，以提高网络利用率。MSS (Maximum Segment Size, 最大报文段长度) 是在 TCP 连接建立时，收发双方协商通信时每一个报文段所能承载的最大数据长度。而 Nagle 算法的目的就是为了尽可能的减少小数据包, 发送 MSS 大小的数据包。

Nagle

　　Nagle算法的基本定义是任意时刻，最多只能有一个未被确认的小段。 所谓“小段”，指的是小于 MSS 尺寸的数据块，所谓“未被确认”，是指一个数据块发送出去后，没有收到对方发送的 ACK。注意: 这里 Nalge 算法在未收到小包的 ACK 仍然能够发送 MSS 大小的数据, 而可发送的数量依赖于上一次 ACK 中附带的接收窗口的大小, 即滑动窗口机制.

Linux 性能分析工具--Valgrind

Valgrind

　　Valgrind 是一款用于内存调试、内存泄漏检测以及性能分析的软件开发工具集. 在使用 C/C++ 开发服务器的时候是不可或缺的分析工具.

功能

　　Valgrind 包含很多的工具 memcheck, cachegrind, helgrind, callgrind 等, 这里主要介绍 memcheck 功能.
　　memcheck 能够探测程序中存在的内存问题, 例如, 使用了未初始化的内存, 读/写已释放的内存, 读/写内存越界, 内存泄漏, 使用 malloc/new/new[] 和 free/delete/delete[] 不匹配, src 和 dst 重叠等等.

TCP 协议分析--CLOSE_WAIT 状态

CLOSE_WAIT

　　CLOSE_WAIT 是在被动关闭连接情况下, 已经接收到了FIN, 但是还没有发送FIN的时刻, 此时连接处于CLOSE_WAIT状态. 通常来讲, CLOSE_WAIT 状态的持续时间应该很短, 正如SYN_RCVD状态. 但是在一些特殊情况下, 就会出现连接长时间处于CLOSE_WAIT状态的情况.
　　主要原因: 在某种情况下对方关闭了socket连接, 但是我方忙于读或者写, 没有关闭连接.
　　一般说来, 如果出现大量的 CLOSE_WAIT 八成是在对端发送了FIN没有正确的处理fd造成的, 仔细检查fd的错误处理应该就能解决了.
